BMP Suite Image List

This document shows the allegedly correct way to interpret the BMP images in BMP Suite, using PNG and JPEG images for reference.

It also shows how your web browser displays the BMP images. But that’s not its main purpose. BMP is poor image format to use on web pages, so it’s probably not very important for a web browser to support it.

File Correct display In your browser Notes
g/pal1.bmp 1 bit/pixel paletted image, in which black is the first color in the palette.
g/pal1wb.bmp 1 bit/pixel paletted image, in which white is the first color in the palette.
g/pal1bg.bmp 1 bit/pixel paletted image, with colors other than black and white.
q/pal1p1.bmp 1 bit/pixel paletted image, with only one color in the palette. The documentation says that 1bpp images have a palette size of 2 (not “up to 2”), but it would be silly for a viewer not to support a size of 1.
g/pal4.bmp Paletted image with 12 palette colors, and 4 bits/pixel.
g/pal4rle.bmp 4-bit image that uses RLE compression.
q/pal4rletrns.bmp
or

or
 An RLE-compressed image that used “delta” codes to skip over some pixels, leaving them undefined. Some viewers make undefined pixels transparent, others make them black, and others assign them palette color 0 (purple, in this case).
g/pal8.bmp Our standard (version 3) paletted image, with 252 palette colors, and 8 bits/pixel.
g/pal8-0.bmp Every field that can be set to 0 is set to 0: pixels/meter=0; colors used=0 (meaning the default 256); size-of-image=0 (this field is largely superfluous).
g/pal8rle.bmp 8-bit image that uses RLE compression.
q/pal8rletrns.bmp
or

or
 8-bit version of q/pal4rletrns.bmp.
g/pal8w126.bmp Images with different widths. In BMP format, rows are padded to a multiple of four bytes, so we test all four possibilities.
g/pal8w125.bmp
g/pal8w124.bmp
q/pal8topdown.bmp BMP images normally are stored from the botttom up, but there is a way to store them from the top down. I have no reason to think it’s invalid to store such an image in a BMP file, but it is unusual.
q/pal8offs.bmp A file with some unused bytes between the palette and the image. This is probably valid, but I’m not 100% sure.
q/pal8oversizepal.bmp An 8-bit image with 300 palette colors. This may be invalid, because the documentation could be interpreted to imply that 8-bit images aren’t allowed to have more than 256 colors.
g/pal8nonsquare.bmp
or
 An image with non-square pixels: the X pixels/meter is twice the Y pixels/meter. Image editors can be expected to leave the image “squashed”; most image viewers really ought to stretch it to the correct proportions.
g/pal8v2.bmp An OS/2-style bitmap, sometimes called “version 2”.
g/pal8v4.bmp A v4 bitmap. I’m not sure that the gamma and chromaticity values in this file are sensible, because I can’t find any detailed documentation of them.
g/pal8v5.bmp A v5 bitmap. Version 5 has additional colorspace options over v4, so it is easier to create, and ought to be more portable.
g/rgb16.bmp A 16-bit image with the default color format: 5 bits each for red, green, and blue, and 1 unused bit. The whitest colors should be (255,255,255), not (248,248,248).
g/rgb16-565.bmp A 16-bit image with a BITFIELDS segment indicating 5 red, 6 green, and 5 blue bits. This is a standard 16-bit format, even supported by old versions of Windows that don’t support any other non-default 16-bit format. The whitest colors should be (255,255,255), not (248,252,248).
q/rgb16-231.bmp An unusual and silly 16-bit image, with 2 red bits, 3 green bits, and 1 blue bit. Most viewers do support this image, but the colors may be darkened with a yellow-green shadow. That’s because they’re doing simple bit-shifting, instead of proper scaling.
q/rgba16-4444.bmp A v5 16-bit image with an alpha channel. There are 4 bits for each color channel, and 4 bits for the alpha channel. It’s not clear if this is valid, but I can’t find anything that suggests it isn’t.
g/rgb24.bmp A perfectly ordinary 24-bit (truecolor) image. If a viewer can’t handle this, it can’t handle anything.
g/rgb24pal.bmp A 24-bit image, with a palette containing 300 colors. There is little if any reason for a truecolor image to contain a palette, but it is legal. The fact that the palette has more than 256 colors may cause some viewers to complain, but the documentation does not mention a size limit.
q/rgb24jpeg.bmp My attempt to make BMP files with embedded JPEG and PNG images. Hardly anything can read them, so maybe I did it wrong.
q/rgb24jpeg.bmp
g/rgb32.bmp A 32-bit image using the default color format for 32-bit images (no BITFIELDS segment). There are 8 bits per color channel, and 8 unused bits. The unused bits are set to 0.
g/rgb32bf.bmp A 32-bit image with a BITFIELDS segment. As usual, there are 8 bits per color channel, and 8 unused bits. But the color channels are in an unusual order, so the viewer must read the BITFIELDS, and not just guess.
q/rgb32fakealpha.bmp Same as g/rgb32.bmp, except that the unused bytes are set to something other than 0. This: is acceptable, but it means the viewer is most likely using heuristics to guess whether the undefined data represents transparency.
q/rgb32-111110.bmp A 32 bits/pixel image, with all 32 bits used: 11 each for the red and green, and 10 for blue. As far as I know, this is perfectly valid, but it is unusual.
q/rgba32.bmp A v5 BMP with an alpha channel. Transparency is barely documented, so it’s possible that this file is not correctly formed. The color channels are in an unusual order, to prevent viewers from passing this test by making a lucky guess.
